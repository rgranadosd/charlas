REGLAS TÉCNICAS CRÍTICAS CPCtelera (verificadas en producción):

INICIALIZACIÓN — ORDEN OBLIGATORIO en game_init():
- cpct_setVideoMode() ANTES de cpct_disableFirmware().
  cpct_setVideoMode() desactiva la lower ROM; solo entonces el patch de
  cpct_disableFirmware() en RAM 0x0038 es efectivo.
  Orden incorrecto → crash silencioso, 0FPS, juego no arranca jamás.
  CORRECTO:
    cpct_setVideoMode(0);
    cpct_disableFirmware();
    cpct_setPalette(...);

TECLADO — usar cpct_scanKeyboard() (NO cpct_scanKeyboard_f()):
- Con firmware deshabilitado, cpct_scanKeyboard_f() causa crash.
- Siempre usar cpct_scanKeyboard() en sistemas con cpct_disableFirmware().

SPRITES — REGLA FUNDAMENTAL: NO llamar playerrender/enemyrender/projectilerender en game_render():
- cpct_drawSprite() copia bytes verbatim a VRAM, sin transparencia.
- En Mode 0, pen 0 = fondo = BLACK (HW 8). El fill interior de los sprites usa pen 0 → invisible contra el fondo negro.
- SIEMPRE usar cpct_drawSolidBox() para dibujar jugador, enemigos y proyectiles.
- Llamar playerrender()/enemyrender()/projectilerender() DESPUÉS de cpct_drawSolidBox borra el solid box: el sprite sobreescribe con pen 0 en el interior → entidad desaparece.
- Para transparencia real, usar cpct_drawSpriteMasked().

SPRITES — tamaño mínimo visible en Mode 0 (160x200):
- Jugador/enemigo: mínimo 4×8 bytes (8×8 px CPC). Recomendado 8×24.
- Proyectil básico: mínimo 4×4 bytes (8×8 px CPC). Menos es invisible.

SPRITES — CODIFICACIÓN DE BYTES EN MODE 0 (CRÍTICO):
- 1 byte = 2 píxeles consecutivos con bits intercalados. NUNCA usar el valor del pen directamente.
- Usar la tabla de conversión pen→byte o cpct_px2byteM0(pen_izquierdo, pen_derecho).
- Los bytes 0x00-0x0F usados como índices directos son INCORRECTOS.

Tabla de valores más usados (un solo pen para ambos píxeles):
  pen0 (BLACK/fondo)     = 0x00    pen1 (BRIGHT_RED/suelo)= 0xC0
  pen2 (BRIGHT_CYAN)     = 0x0C    pen3 (BRIGHT_BLUE)    = 0xCC
  pen4 (CYAN/enemigo)    = 0x30    pen5 (BRIGHT_GREEN)   = 0xF0
  pen6 (BRIGHT_WHITE/player) = 0x3C  pen7 =(varios)      = 0xFC

⚠ pen1+pen1 = 0xC0 (BLACK), NO 0x03 (eso es pen8+pen8 = BRIGHT_BLUE)

Tabla completa pen_izquierdo × pen_derecho → byte hex (filas = pen izq, cols = pen der):
       P0   P1   P2   P3   P4   P5   P6   P7
  P0: 0x00 0x80 0x08 0x88 0x20 0xA0 0x28 0xA8
  P1: 0x40 0xC0 0x48 0xC8 0x60 0xE0 0x68 0xE8
  P2: 0x04 0x84 0x0C 0x8C 0x24 0xA4 0x2C 0xAC
  P3: 0x44 0xC4 0x4C 0xCC 0x64 0xE4 0x6C 0xEC
  P4: 0x10 0x90 0x18 0x98 0x30 0xB0 0x38 0xB8
  P5: 0x50 0xD0 0x58 0xD8 0x70 0xF0 0x78 0xF8
  P6: 0x14 0x94 0x1C 0x9C 0x34 0xB4 0x3C 0xBC
  P7: 0x54 0xD4 0x5C 0xDC 0x74 0xF4 0x7C 0xFC

Ejemplo de sprite correcto (4 bytes ancho, fill=pen6=bright_white, contorno=pen0=black):
  Fila contorno: 0x00, 0x00, 0x00, 0x00  (pen0+pen0 = borde negro)
  Fila relleno:  0x14, 0x3C, 0x3C, 0x28  (pen0|pen6, pen6+pen6, pen6|pen0)
  Fila relleno:  0x14, 0x3C, 0x3C, 0x28
  Fila contorno: 0x00, 0x00, 0x00, 0x00
⚠ En la práctica, usar cpct_drawSolidBox con cpct_px2byteM0(6,6) = 0x3C, NO cpct_drawSprite.

TECLADO — Compatibilidad con Caprice32 (joystick_emulation=1):
- Caprice32 en macOS con joystick_emulation=1 envía las flechas del PC al puerto joystick del CPC.
- SIEMPRE incluir Joy0_Left/Right/Up/Down/Fire1 en input_update() como alternativa a cursor keys.
- Ejemplo correcto de input_update():
    cpct_scanKeyboard();
    left  = (u8)(cpct_isKeyPressed(Key_CursorLeft)  || cpct_isKeyPressed(Key_A) || cpct_isKeyPressed(Joy0_Left));
    right = (u8)(cpct_isKeyPressed(Key_CursorRight) || cpct_isKeyPressed(Key_D) || cpct_isKeyPressed(Joy0_Right));
    jump  = (u8)(cpct_isKeyPressed(Key_CursorUp)    || cpct_isKeyPressed(Key_W) || cpct_isKeyPressed(Joy0_Up));
    shoot = (u8)(cpct_isKeyPressed(Key_Space)       || cpct_isKeyPressed(Key_X) || cpct_isKeyPressed(Joy0_Fire1));

cpct_drawSolidBox — LÍMITE CRÍTICO: ancho máximo = 64 bytes (válido en Mode 0 Y Mode 1):
- La función usa 63 LDIs con JR dinámico. width > 64 → JR negativo → NO escribe nada.
- Pantalla CPC = 80 bytes ancho en TODOS los modos gráficos. NUNCA llamar con width=80 en ningún modo.
- Para borrar la pantalla completa (80 bytes) SIEMPRE dividir en dos llamadas de 40:
    cpct_drawSolidBox(CPCT_VMEM_START,                              color, 40, 200);
    cpct_drawSolidBox(cpct_getScreenPtr(CPCT_VMEM_START, 40, 0),   color, 40, 200);
- NUNCA usar cpct_clearScreen_f8() en el game loop: tarda ~21ms (> 1 frame a 50Hz CPC).
  El haz CRT ve pantalla en blanco durante ese tiempo → flickering total garantizado.
  Solo usar para pantallas estáticas (menú, game over) fuera del loop.
- En el game loop usar PATRÓN ERASE-REDRAW:
    1. Borrar sprites en su posición ANTERIOR: cpct_drawSolidBox(prev_pos, 0x00, w, h)
    2. tilemap_render() — restaura fondo sucio
    3. Dibujar sprites en posición ACTUAL (jugador primero, luego enemigos)
    4. Guardar posición actual en prev_x/prev_y para el siguiente frame
- Para cualquier draw parcial donde ancho > 40, dividir igualmente en dos llamadas de ≤40.

ENTRY POINT — JP STUB obligatorio en game.c:
- game.c se enlaza ANTES que main.c (orden alfabético). El código en 0x4000 es game.c, NO cpc_run_address.
- La PRIMERA función de game.c (antes de cualquier otra) DEBE ser:
    void __game_entry_jp(void) __naked {
        __asm
            .globl cpc_run_address
            jp cpc_run_address
        __endasm;
    }
- DISC.BAS SIEMPRE usa CALL 16384. NUNCA la dirección exec_addr calculada.

COORDENADAS — LÍMITES DE PANTALLA EN MODE 0:
- cpct_getScreenPtr(base, x, y): x en BYTES (0-79), y en filas (0-199).
- Spawn de entidades: spawn_x + sprite_w ≤ 79. Clámpa siempre: if (spawn_x > 74) spawn_x = 74;
- Y inicial del jugador: player.y = tilemap_ground_y() - player.h (pies al ras del suelo).
