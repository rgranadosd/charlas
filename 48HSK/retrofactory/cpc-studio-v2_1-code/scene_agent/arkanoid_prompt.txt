Create a complete Arkanoid-style breakout game in CPCtelera for Amstrad CPC using Mode 0.

0. SCREEN LAYOUT (MANDATORY — zones are disjoint and must NEVER overlap):
   - Mode 0 coordinates: x in BYTES 0..79, y in PIXELS 0..199. Each text character is 4 bytes wide and 8 pixels tall.
   - HUD ZONE: rows y=0..7. ONLY the four HUD texts defined in section 1 live here. Nothing else is ever drawn here, and no moving entity ever enters here.
   - PLAY AREA: rows y=8..199. Ball, bricks and paddle exist ONLY inside the play area.
   - Bricks: 5 rows x 10 columns at y=20..59. The CELL pitch is BLOCK_WIDTH=8 bytes by BLOCK_HEIGHT=8 pixels; brick (i,j) occupies the cell whose top-left is x=j*BLOCK_WIDTH, y=20+i*BLOCK_HEIGHT. To make bricks look like SEPARATE blocks (never solid full-width stripes), DRAW each brick as a solid box of (BLOCK_WIDTH-1) bytes by (BLOCK_HEIGHT-1) pixels at that top-left — leaving a 1-byte / 1-pixel gutter. Collision and erase still use the FULL BLOCK_WIDTH x BLOCK_HEIGHT cell.
   - Paddle: PADDLE_Y=190, PADDLE_WIDTH=10 bytes, PADDLE_HEIGHT=6 pixels (it is a solid BAR, not a thin line). Ball: BALL_W=2 bytes, BALL_H=4 pixels. FLOOR_Y=195.

1. HUD (MANDATORY):
   - Draw labels AND initial values ONCE in init_game() using EXACTLY this code:
       pv = cpct_getScreenPtr(CPCT_VMEM_START, 0, 0);
       cpct_drawStringM0("LIVES:", pv, 1, 0);
       pv = cpct_getScreenPtr(CPCT_VMEM_START, 40, 0);
       cpct_drawStringM0("SCORE", pv, 1, 0);
     then draw the initial lives digit at x=24,y=0 and the initial 3-digit score at x=64,y=0.
   - During the game, redraw ONLY the digits, ONLY when their value changes: lives digit at x=24, 3-digit score at x=64.
   - The complete list of HUD draws is: x=0 (label), x=24 (digit), x=40 (label), x=64 (3 digits). FORBIDDEN: any other text or element in the HUD zone (no LEVEL, no HIGH SCORE, no title, no branding). FORBIDDEN: local char arrays for labels — string literals only; digits may use a small null-terminated buffer.

2. GAME STATE:
   - Globals: g_lives (3 at start), g_score (0 at start), game_over=0, ball_launched=0, blocks_remaining=50, u8 block_grid[5][10], paddle_x, ball_x, ball_y, i8 ball_vx, i8 ball_vy, prev_ball_x, prev_ball_y, prev_paddle_x.
   - All state variables shared between functions are GLOBAL. NEVER shadow a global with a local of the same name.
   - init_game() MUST explicitly assign every state variable exactly once (no duplicated init blocks): g_lives=3, g_score=0, game_over=0, ball_launched=0, blocks_remaining=50, all block_grid cells=1, paddle_x=35, prev_paddle_x=35, ball_x=paddle_x+(PADDLE_WIDTH/2), ball_y=PADDLE_Y-BALL_H, ball_vx=1, ball_vy=-1, prev_ball_x=ball_x, prev_ball_y=ball_y.
   - reset_ball() MUST set: ball_x=paddle_x+(PADDLE_WIDTH/2), ball_y=PADDLE_Y-BALL_H, ball_vx=1, ball_vy=-1, ball_launched=0, prev_ball_x=ball_x, prev_ball_y=ball_y.
   - No state machine enums. Flags only: game_over, ball_launched.

3. PHYSICS (MANDATORY — implement EXACTLY these steps and ADD NOTHING ELSE.
   In particular: NEVER reposition ball_x/ball_y after a collision except where this section explicitly says so):
   - Ball speed is constant: ball_vx is 1 or -1, ball_vy is 1 or -1 — never 0, never ±2.
   - While NOT launched: every frame set ball_x = paddle_x + PADDLE_WIDTH/2 and ball_y = PADDLE_Y - BALL_H. Launch with Space: set ball_launched=1.
   - Side walls: if moving right and ball_x reaches 80-BALL_W, invert ball_vx. If moving left and ball_x reaches 0, invert ball_vx. Only ball_vx changes.
   - Ceiling (top of PLAY AREA, not of the screen): if (ball_vy < 0 && ball_y <= 8) { ball_y = 8; ball_vy = 1; } — the ball NEVER enters rows y=0..7. Only ball_vy changes.
   - Brick collision: iterate the grid; check `if (block_grid[i][j])` BEFORE the AABB test. On hit do EXACTLY: block_grid[i][j]=0; blocks_remaining--; g_score += 10; ball_vy = -ball_vy; erase that brick from screen (cpct_drawSolidBox with colour 0x00 at the brick position); play SFX_BRICK_HIT; then `goto block_done;` so AT MOST ONE brick is destroyed per frame. Put the label `block_done:;` right after the outer loop. Do NOT touch ball_x or ball_y here.
   - Paddle collision: if ball moving down overlaps the paddle (AABB), do EXACTLY: ball_vy = -ball_vy; ball_y = PADDLE_Y - BALL_H; play SFX_PADDLE_HIT. (This is the ONLY collision allowed to set ball_y.) NEVER hardcode ball_vy = -1 — always use ball_vy = -ball_vy.
   - Floor: the check goes INSIDE the downward-movement branch: when moving down, compute ball_bottom = ball_y + BALL_H - 1 AFTER moving; if ball_bottom >= FLOOR_Y then g_lives--, play SFX_LIFE_LOST, and either reset_ball() (lives left) or set game_over=1 and play SFX_GAME_OVER. NEVER clamp ball_y at any maximum — only this floor check handles the bottom.

4. RENDERING OWNERSHIP (MANDATORY):
   - init_game() draws ONCE: cpct_clearScreen(0), the full brick grid, the HUD labels and initial values, AND the paddle at (paddle_x, PADDLE_Y) and the ball at (ball_x, ball_y) so BOTH are visible before the first key press. (A game that only draws the paddle/ball once they move is WRONG.)
   - prev_ball_x, prev_ball_y and prev_paddle_x are the "previous frame" positions used ONLY to erase. They are updated in EXACTLY ONE place: inside draw_game(), AFTER erasing the old position and drawing the new one. The physics/update step moves ball_x/ball_y/paddle_x but MUST NEVER read or assign prev_*. (If prev_* is set before draw_game runs, the erase clears the wrong cell and the entity leaves a TRAIL across the screen — this is the #1 bug to avoid.)
   - draw_game(), every frame, in this exact order:
       (a) BALL: erase at (prev_ball_x, prev_ball_y) with colour 0 (BALL_W x BALL_H); draw at (ball_x, ball_y) with the ball colour; THEN prev_ball_x=ball_x; prev_ball_y=ball_y;
       (b) PADDLE: erase at (prev_paddle_x, PADDLE_Y) with colour 0 (PADDLE_WIDTH x PADDLE_HEIGHT — a solid bar PADDLE_HEIGHT px tall, NEVER 1-2 px); draw at (paddle_x, PADDLE_Y) with the paddle colour; THEN prev_paddle_x=paddle_x;
     Always erase+draw every frame — do NOT skip with an `if (prev != cur)` guard (re-erasing the same cell is harmless and guarantees no trail and a permanently visible entity).
   - HUD digits: redraw ONLY when the value changed.
   - Bricks are erased individually at the moment of destruction (inside the collision step). FORBIDDEN: redrawing the brick grid, the HUD labels, or any full scene inside the main loop — it destroys the frame budget on a 4MHz Z80.
   - Colours via cpct_px2byteM0(pen,pen) assigned in init_game() — NEVER hardcode raw colour bytes. Distinct colour per brick row. Palette via cpct_setPALColour; cpct_setBorder(HW_BLACK).

5. LEVEL FLOW:
   - When blocks_remaining reaches 0: refill block_grid with 1s, blocks_remaining=50, redraw the brick grid once, reset_ball(). g_score and g_lives KEEP their values — they are NEVER reset during a session.
   - When game_over becomes 1: stop updating the game and draw "GAME OVER" once, centered inside the play area (e.g. x=30, y=100). No start screen, no separate game-over screen, no level-complete screen or message.

6. AUDIO:
   - The audio subsystem is implemented by the AUDIO SPECIALIST AGENT in src/audio.h + src/audio.c (AY-3-8912 beep backend, no external assets). It must expose: audio_init(), audio_update(), audio_play_sfx(u8 sfx_id) and exactly these SFX ids: SFX_WALL_HIT, SFX_PADDLE_HIT, SFX_BRICK_HIT, SFX_LIFE_LOST, SFX_GAME_OVER.
   - The gameplay code (src/main.c) NEVER implements audio — it only includes "audio.h" and calls:
     audio_init() once in init_game(); audio_update() at the end of every main-loop iteration;
     audio_play_sfx(SFX_WALL_HIT) on wall/ceiling bounce, SFX_PADDLE_HIT on paddle bounce, SFX_BRICK_HIT on brick destruction, SFX_LIFE_LOST when losing a life, SFX_GAME_OVER when lives reach zero.

7. TECHNICAL REQUIREMENTS:
   - C89/SDCC: declare ALL variables at the top of each function, before any statement.
   - No stdio.h, stdlib.h, printf, puts, sprintf. CPCtelera API only (<cpctelera.h>).
   - Keyboard: cpct_scanKeyboard_f() once per loop; keys are Key_Space, Key_CursorLeft, Key_CursorRight (NEVER KEY_Space or CPCT_KEY_*). Read Space exactly ONCE (no duplicated Space block).
   - Paddle movement uses a fixed step PADDLE_STEP=2 and MUST stay clamped within 0..(80-PADDLE_WIDTH) at both ends (apply the safe u8 movement pattern).
   - cpct_waitVSYNC() once per main-loop iteration.
   - Single gameplay file: src/main.c.

8. INVARIANTS (the final code must satisfy ALL of these — they are checkable by reading the code):
   - At most ONE brick is destroyed per frame (the goto guarantees it).
   - ball_y is never smaller than 8 once the game runs (HUD zone is untouchable).
   - g_score and g_lives are assigned their starting values in init_game() ONLY — no other assignment except g_score += 10 and g_lives--.
   - The only drawStringM0 destinations in the HUD zone are x=0, x=24, x=40, x=64 at y=0 (plus the single GAME OVER text inside the play area).
   - Every draw of a moving entity in the main loop is preceded by erasing its previous position.
   - prev_ball_x, prev_ball_y and prev_paddle_x are ASSIGNED in exactly one place — inside draw_game(), AFTER the erase+draw — and are NEVER assigned inside update_game()/the physics step. (Guarantees no trail.)
   - init_game() draws the paddle and the ball at least once, so both are visible before any key is pressed.
   - Each brick is drawn (BLOCK_WIDTH-1) x (BLOCK_HEIGHT-1) so a gutter separates it from its neighbours; the brick grid is NEVER rendered as solid full-width stripes.
   - paddle_x always stays within 0..(80-PADDLE_WIDTH) and never wraps around (no u8 underflow at the left edge).
   - The collision steps contain no statements beyond those listed in section 3.
   - The paddle is drawn as a solid box of PADDLE_WIDTH x PADDLE_HEIGHT (6 px tall), NEVER a 1 or 2 px line.
   - C89: every variable (including static) is declared at the TOP of its function, before any statement — no declaration appears after a statement.
