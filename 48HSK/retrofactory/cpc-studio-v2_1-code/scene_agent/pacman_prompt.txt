Create a simple Pac-Man-style maze game in CPCtelera for Amstrad CPC using Mode 0.

0. SCREEN LAYOUT (MANDATORY — zones are disjoint and must NEVER overlap):
   - Mode 0 coordinates: x in BYTES 0..79, y in PIXELS 0..199. Each text character is 4 bytes wide and 8 pixels tall.
   - HUD ZONE: rows y=0..7. ONLY the two HUD texts defined in section 1 live here. Nothing else is ever drawn here, and no moving entity ever enters here.
   - PLAY AREA: rows y=8..199. The maze, the player and the ghost exist ONLY inside the play area.
   - Maze grid: MAZE_ROWS=11 rows x MAZE_COLS=19 columns of tiles. Each tile is TILE=4 bytes wide and 16 pixels tall, starting at y=16. A tile is either a WALL or an open CORRIDOR; open corridors contain a dot until eaten.
   - MAZE LAYOUT (MANDATORY — the maze MUST be this exact map, NOT a generated/empty box). Hardcode it as a static array of 11 string literals of 19 chars each, '#'=wall '.'=corridor, and copy it into g_maze (g_maze[i][j]=1 if '#', else 0) in init_game(). The interior MUST contain internal walls exactly as drawn — never just a hollow rectangle:
       "###################"
       "#........#........#"
       "#.###.##.#.##.###.#"
       "#.................#"
       "#.###.#.###.#.###.#"
       "#...#...#.#...#...##"
       "#.#.#.#.###.#.#.#..#"
       "#.................#"
       "#.###.##.#.##.###.#"
       "#........#........#"
       "###################"
   - Player starts at tile (row=9, col=9) and the ghost at (row=1, col=1) — both are corridor tiles in the map above. A dot exists on every corridor tile except the two starting tiles.

1. HUD (MANDATORY):
   - Draw labels AND initial values ONCE in init_game() using EXACTLY this code:
       pv = cpct_getScreenPtr(CPCT_VMEM_START, 0, 0);
       cpct_drawStringM0("SCORE", pv, 1, 0);
       pv = cpct_getScreenPtr(CPCT_VMEM_START, 48, 0);
       cpct_drawStringM0("LIVES:", pv, 1, 0);
     then draw the initial 3-digit score at x=24,y=0 and the initial lives digit at x=72,y=0.
   - During the game, redraw ONLY the digits, ONLY when their value changes: 3-digit score at x=24, lives digit at x=72.
   - The complete list of HUD draws is: x=0 (label), x=24 (3 digits), x=48 (label), x=72 (digit). FORBIDDEN: any other text or element in the HUD zone (no LEVEL, no title, no branding). FORBIDDEN: local char arrays for labels — string literals only; digits may use a small null-terminated buffer.

2. GAME STATE:
   - Globals: g_score (0 at start), g_lives (3 at start), game_over=0, dots_remaining, player_x, player_y, prev_player_x, prev_player_y, player_dir, ghost_x, ghost_y, prev_ghost_x, prev_ghost_y, u8 maze[MAZE_ROWS][MAZE_COLS] (1=wall, 0=corridor), u8 dots[MAZE_ROWS][MAZE_COLS] (1=dot present).
   - Player and ghost positions are TILE indices (column, row), not pixels. Convert to screen with x_byte = col*TILE, y_pix = 16 + row*16.
   - TILE→SCREEN CONVERSION (MANDATORY — applies to maze walls, dots, player AND ghost, every draw and erase): the X byte is `col*TILE`; the Y pixel is `16 + row*16`. NEVER use `row*TILE` (or any `row*4`) for the Y coordinate — that compresses the board into the top ~44px and DRAWS OVER THE HUD ZONE (y=0..7), corrupting SCORE/LIVES. Tiles are 16 pixels tall; rows are 16px apart and the board starts at y=16. Every cpct_getScreenPtr for a tile uses (col*TILE, 16 + row*16).
   - WALL TILE SIZE (MANDATORY): each WALL tile is drawn as a solid box that FILLS THE WHOLE CELL: width = TILE bytes, height = 16 pixels → `cpct_drawSolidBox(pv, wall_color, TILE, 16)`. NEVER draw the wall as TILE×TILE (that is only 4px tall → thin sliver walls with gaps, the maze looks empty). The player and ghost are drawn TILE×16 as well (full cell). Erasing a moving entity also erases a TILE×16 box.
   - All state variables shared between functions are GLOBAL. NEVER shadow a global with a local of the same name.
   - init_game() MUST explicitly assign every state variable exactly once (no duplicated init blocks): g_score=0, g_lives=3, game_over=0; copy g_maze from the MANDATORY map in section 0 (NEVER fill it procedurally with a hollow rectangle); set a dot on every corridor tile except the player/ghost start tiles and count them into dots_remaining; place player at (9,9) and ghost at (1,1); prev_* = current positions, player_dir = stopped.
   - reset_positions() MUST set player and ghost back to their start tiles and update prev_* accordingly, WITHOUT resetting g_score, g_lives, the maze or the dots.
   - No state machine enums. Flags only: game_over.

3. MOVEMENT & RULES (MANDATORY — implement EXACTLY these steps and ADD NOTHING ELSE.
   In particular: NEVER move a character onto a WALL tile, and NEVER move more than one tile per frame):
   - Read cursor keys; they set player_dir (up/down/left/right). The player moves ONE tile per frame in player_dir ONLY IF the destination tile is a corridor (maze cell == 0); otherwise it stays put. NEVER let the player pass through a wall.
   - KEYBOARD READING (MANDATORY — calling cpct_scanKeyboard_f() is NOT enough; you MUST actually TEST each key): after cpct_scanKeyboard_f(), read the four cursor keys with cpct_isKeyPressed and set player_dir, e.g.:
        if      (cpct_isKeyPressed(Key_CursorUp))    player_dir = DIR_UP;
        else if (cpct_isKeyPressed(Key_CursorDown))  player_dir = DIR_DOWN;
        else if (cpct_isKeyPressed(Key_CursorLeft))  player_dir = DIR_LEFT;
        else if (cpct_isKeyPressed(Key_CursorRight)) player_dir = DIR_RIGHT;
     The file MUST contain at least one cpct_isKeyPressed(Key_Cursor...) call for EACH of the four directions. A build that scans the keyboard but never calls cpct_isKeyPressed is WRONG (the player can never move).
   - Dot eating: when the player enters a tile whose dot is present, set its dot to 0, g_score += 10, dots_remaining--, and erase the dot visually. AT MOST ONE dot is eaten per frame (it is the player's single destination tile).
   - Ghost moves ONE tile per frame toward the player (greedy: reduce row or column distance) but NEVER onto a wall tile; if blocked, it may stay. NEVER move the ghost more than one tile per frame. To compare row/column distance, do NOT use abs() (it does not exist without stdlib) — compute each absolute difference inline with an if (e.g. dr = (player_row>=ghost_row) ? player_row-ghost_row : ghost_row-player_row;).
   - Collision: if player and ghost occupy the same tile, g_lives--, then either reset_positions() (lives left) or set game_over=1.
   - When dots_remaining reaches 0: refill all dots, recount dots_remaining, reset_positions(). g_score and g_lives KEEP their values — they are NEVER reset during a session.

4. RENDERING OWNERSHIP (MANDATORY):
   - init_game() draws ONCE: cpct_clearScreen(0), the full maze (walls + initial dots), the HUD labels and initial values.
   - The main loop draws ONLY: (a) player erase/draw using prev_player_x/prev_player_y, (b) ghost erase/draw using prev_ghost_x/prev_ghost_y, (c) HUD digits when their value changed. Always update prev_* after drawing.
   - Dots are erased individually at the moment they are eaten. FORBIDDEN: redrawing the maze, the dots, the HUD labels, or any full scene inside the main loop — it destroys the frame budget on a 4MHz Z80.
   - Colours via cpct_px2byteM0(pen,pen) assigned in init_game() — NEVER hardcode raw colour bytes. Walls, dots, player and ghost use distinct pens. Palette via cpct_setPALColour; cpct_setBorder(HW_BLACK).

5. STATE FLOW:
   - When game_over becomes 1: stop updating and draw "GAME OVER" once, centered inside the play area (e.g. x=30, y=100). No start screen, no separate game-over screen.

6. AUDIO:
   - The audio subsystem is implemented by the AUDIO SPECIALIST AGENT in src/audio.h + src/audio.c (AY-3-8912 beep backend, no external assets). It must expose: audio_init(), audio_update(), audio_play_sfx(u8 sfx_id) and exactly these SFX ids: SFX_DOT_EAT, SFX_DEATH, SFX_GAME_OVER.
   - The gameplay code (src/main.c) NEVER implements audio — it only includes "audio.h" and calls:
     audio_init() once in init_game(); audio_update() at the end of every main-loop iteration;
     audio_play_sfx(SFX_DOT_EAT) when a dot is eaten, SFX_DEATH when the ghost catches the player, SFX_GAME_OVER when lives reach zero.

7. TECHNICAL REQUIREMENTS:
   - C89/SDCC: declare ALL variables at the top of each function, before any statement.
   - No stdio.h, stdlib.h, printf, puts, sprintf. CPCtelera API only (<cpctelera.h>). NEVER call abs(), min(), max() or any stdlib function — compute them inline with if-comparisons.
   - Keyboard: cpct_scanKeyboard_f() once per loop; keys are Key_CursorUp, Key_CursorDown, Key_CursorLeft, Key_CursorRight (NEVER KEY_* or CPCT_KEY_*).
   - cpct_waitVSYNC() once per main-loop iteration.
   - Single gameplay file: src/main.c.

7b. FORBIDDEN CROSS-GAME CODE (this is a MAZE game — it has NO paddle, NO ball, NO bricks/blocks):
   - NEVER declare, assign, read or reference ANY of these identifiers (they belong to a different game and DO NOT exist here): ball_x, ball_y, ball_vx, ball_vy, ball_launched, BALL_W, BALL_H, paddle_x, prev_paddle_x, PADDLE_WIDTH, PADDLE_Y, PADDLE_HEIGHT, blocks_remaining, block_grid, BLOCK_WIDTH, BLOCK_HEIGHT, BLOCK_ROWS, BLOCK_COLS, FLOOR_Y, g_level, reset_ball().
   - init_game() assigns EXACTLY the globals listed in section 2 GAME STATE and NOTHING ELSE. There is exactly ONE assignment of g_score, g_lives and game_over in the whole program (inside init_game). Do NOT add a second/duplicate "reset state" block, and do NOT add a generic state-reset block copied from any other game.
   - If you are about to write a variable that is not declared in section 2, STOP — it is wrong. Every assignment in init_game must correspond to a variable from section 2.

8. INVARIANTS (the final code must satisfy ALL of these — they are checkable by reading the code):
   - NONE of the forbidden cross-game identifiers in section 7b appear anywhere in the file.
   - init_game() contains exactly one assignment block; g_score/g_lives/game_over are each assigned exactly once.
   - The player NEVER moves onto a wall tile (every move is guarded by a maze cell == 0 check).
   - The ghost NEVER moves onto a wall tile.
   - Each character moves AT MOST one tile per frame.
   - AT MOST one dot is eaten per frame.
   - g_score and g_lives are assigned their starting values in init_game() ONLY — no other assignment except g_score += 10 and g_lives--.
   - The only drawStringM0 destinations in the HUD zone are x=0, x=24, x=48, x=72 at y=0 (plus the single GAME OVER text inside the play area).
   - Every draw of a moving entity in the main loop is preceded by erasing its previous position.
   - The maze and dots are drawn once in init_game(), never redrawn wholesale in the main loop.
   - g_maze is initialized by copying the exact MANDATORY map in section 0 (with internal walls), NEVER as a hollow border-only rectangle.
   - Every tile draw/erase (maze, dots, player, ghost) uses Y = 16 + row*16. NO draw of a tile uses row*TILE (row*4) for its Y coordinate, and NOTHING is ever drawn into the HUD zone y=0..7 except the HUD texts.
   - Every WALL tile is drawn as a FULL cell: cpct_drawSolidBox(pv, wall_color, TILE, 16). NO wall is drawn as TILE×TILE (4px tall). Player/ghost boxes are TILE×16 too.
   - The file contains at least one cpct_isKeyPressed(Key_CursorUp), one (Key_CursorDown), one (Key_CursorLeft) and one (Key_CursorRight) call, and they set player_dir. Calling cpct_scanKeyboard_f() without any cpct_isKeyPressed is a violation.
